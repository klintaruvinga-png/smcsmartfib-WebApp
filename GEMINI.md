# SMC SuperFIB Dashboard - Gemini CLI Instructions

This repository is the SMC SuperFIB Dashboard / MT5 EA migration project. This file provides the foundational mandates for Gemini CLI operation within this workspace.

## Core Mandates & Priorities
- **Read First:** Always start by reviewing `AGENTS.md`, `CONTEXT.md`, and `docs/agents/skill-index.md`.
- **Visual Authority:** The Pine indicator (`SMC_SuperFib_v13.1.3.pine`) is the visual authority. Backend and EA logic must match Pine behavior (Parity Gate).
- **Evidence-Based:** Never claim a test or verification passed without running the command and citing the output.
- **Autonomous PRs:** After any file-changing session, Gemini must create a GitHub PR using `gh pr create --fill`.

## Domain Context (from CONTEXT.md)
- **Authority Fib (AF):** The primary reference levels generated by Pine.
- **Parity Gate:** The checkpoint where Pine and MT5/backend fib generation must match.
- **Market Stream:** The EA-to-backend data feed route.
- **Plan Card:** Dashboard UI component for next-best blueprints.
- **Avoid Language:** Do not call Pine behavior "legacy" or backend fibs "correct" until parity is proven.

## Agent Skills & Workflows
Follow the workflows defined in `docs/agents/skill-index.md`. Key skills include:
- `diagnose`: Broad problem discovery and root-cause analysis across layers.
- `tdd`: Intentional code/test cycles for parity and display logic.
- `pine-mt5-fib-parity`: Focused parity validation between Pine and MT5/Backend.
- `ea-backend-bridge`: Debugging market-stream, heartbeats, and sync logic.
- `dashboard-plan-cards`: UI/UX work on plan cards and currency formatting.
- `to-prd`: Preparing a production-ready patch with full validation.
- `handoff`: Final session summary with branch/PR links and validation evidence.

## Engineering Standards
- **Branching:** Use descriptive feature branches (e.g., `feat/parity-fix`, `fix/heartbeat-sync`). Never work directly on `main`.
- **Testing:** Prioritize small, testable vertical slices. Use scripts in `scripts/` (e.g., `parity-validator.php`, `run-phase4-parity.ps1`) for validation.
- **Documentation:** For docs-only work, explain what was checked and why no runtime command was available.

## PR Protocol
Every PR must include:
- One paragraph summarizing the issue.
- Root cause identified.
- Exact files changed and what changed in each.
- Regression protections added.
- Parity impact (Pine / MT5 / Backend / Dashboard).
- Systems intentionally not touched (Do Not Touch list).

### PR Review Fix Stage
If a PR has review comments (P1/P2/P3), read `reports/codex-review.json` and follow the remediation rules in `.github/prompts/claude-remediate-prompt.md`.
- **P1/P2:** Must be addressed.
- **P3:** Requires evidence; skip with reason if absent.
- Never widen scope or ask for permission.

## Workflow State
- Preserve `.smc-workflow-state.json` and read it before making edits.
- Use `reports/` for local planning and pipeline artifacts.
- Do not modify workflow state or reports without cause.
